# 🎓 FPMI RAG Bot

Телеграм-бот с искусственным интеллектом, который отвечает на вопросы 
о Факультете прикладной математики и информатики (ФПМИ) БГУ.
Бот использует архитектуру **RAG (Retrieval-Augmented Generation)**: 
он ищет актуальную информацию на официальном сайте факультета и формирует ответ с помощью LLM (Llama 3.1).

## 🚀 Быстрый старт (macOS / Linux)

Для запуска требуется установленный **Docker** и **Docker Compose**. \
Важно отметить, что первый запуск может идти достаточно долго. Связанно это в основном с загрузкой всех зависимостей и индексацией данных
Если вы хотите, то можно использовать локально запущенную Ollama (так она должна работать быстрее, чем Docker образ)

### 1. Настройка окружения

1.  Склонируйте репозиторий.
2.  Создайте файл `.env` на основе примера:
    ```bash
    cp .env.example .env
    ```
3.  Откройте `.env` и вставьте токен вашего бота в `TGSERVER__TOKEN`. Остальные настройки можно оставить по умолчанию.
    также, если хочется просмотреть на то, как проходит запрос, сколько времени на все затрачивается и так далее, 
    можно воспользоваться LangSmith (ссылки [тут](#полезные-ссылки)). Вам будет необходимо перейти на их сайт, получить ключ и добавить его в поле `LANGCHAIN_API_KEY` 
4. Для настройки локальной модели: запустите Ollama, в терминале введите `export OLLAMA_HOST=0.0.0.0` и `ollama serve`. в `.env` установите `OLLAMA_HOST=http://host.docker.internal:11434`

### 2. Автоматическая установка

Запустите интерактивный скрипт. Он сам проверит доступность Ollama на хосте, предложит переключиться на Docker при необходимости, проведет миграции и индексацию.

```bash
chmod +x setup.sh
./setup.sh
```

### 3. Подключение к Telegram (Localtunnel)

Так как бот работает локально, Telegram не может отправлять ему сообщения напрямую. Нужно создать туннель.

1.  Откройте **новое окно терминала**.
2.  Установите и запустите `localtunnel` (требуется Node.js):
    ```bash
    npx localtunnel --port 8080 --subdomain famcs-answer-bot
    ```
    *(Если поддомен занят, уберите флаг `--subdomain` и используйте случайный адрес).*

3.  Скопируйте полученную ссылку (например, `https://famcs-answer-bot.loca.lt`).
4.  Откройте файл `.env` и обновите переменную `TGSERVER__WEBHOOK_URL`.
    **ВАЖНО:** Обязательно добавьте `/webhook` в конце ссылки!
    ```ini
    TGSERVER__WEBHOOK_URL=https://famcs-answer-bot.loca.lt/webhook
    ```
5.  Перезапустите контейнер бота, чтобы применить новый адрес:
    ```bash
    docker-compose restart bot
    ```

🎉 **Готово!** Можно писать боту `/start` и задать ему свой первый вопрос: "Кто является деканом факультета в настоящее время?". \
Чтобы остановить бота, введи `docker-compose down -v`

#### Важно

Бот может отвечать не быстро, поэтому -- следите за его состоянием в логах

## 🛠 Ручной запуск (если скрипт не сработал)

Если вы хотите контролировать каждый шаг, выполните команды последовательно (так же хорошим примером послужит сам файл [setup.sh](setup.sh)):

1.  **Запуск сервисов:**
    ```bash
    docker-compose up -d --build
    ```
     *Примечание: По умолчанию контейнер `ollama` не запустится (используется профиль).*

2.  **Запуск LLM (выберите один вариант):**
    *   **Вариант А: Внутри Docker (медленно)**
        ```bash
        # Запуск контейнера
        docker-compose --profile ollama-docker up -d ollama
        # Загрузка модели
        docker-compose --profile ollama-docker exec ollama ollama pull llama3.1
        ```
    *   **Вариант Б: Нативно на вашем компьютере (быстро)**
        Запустите приложение Ollama, в терминале введите `export OLLAMA_HOST=0.0.0.0 && ollama serve`, а в `.env` укажите `OLLAMA_HOST=http://host.docker.internal:11434`.

3.  **Миграции базы данных:**
    ```bash
    docker-compose exec rag-cli alembic upgrade head
    ```

4.  **Индексация данных (Сбор информации с сайта):**
    Без этого шага бот не будет знать ответов.
    ```bash
    docker-compose exec rag-cli python main.py index test
    ```

## 🧠 Функционал и Архитектура

Проект реализует полный цикл RAG-системы:

1.  **Indexing (Индексация):**
    *   Парсинг HTML страниц (поддержка списков, таблиц, статей).
    *   Очистка от шума (меню, футеры).
    *   Разбиение на чанки (Semantic Chunking).
    *   Создание векторных эмбеддингов (модель `intfloat/multilingual-e5-large`).
    *   Сохранение в **ChromaDB** (векторный поиск) и **BM25** (поиск по ключевым словам).

2.  **Retrieval (Поиск):** 
    *   Гибридный поиск (Vector + Keyword).
    *   **Query Expansion:** LLM генерирует синонимы вопроса для лучшего поиска.
      * для скорости работы пока отключен
    *   **Re-ranking:** Пересортировка найденных документов с помощью Cross-Encoder для максимальной релевантности.
      *  для скорости работы пока отключен

3.  **Generation (Генерация):**
    *   LLM формирует финальный ответ на основе найденного контекста.

4.  **Telegram Bot:**
    *   Асинхронный бот на **aiogram 3.x**.
    *   Сохранение истории вопросов и ответов в **PostgreSQL**.
    *   Dependency Injection контейнер для управления зависимостями.
5. **Conversational Memory:**
    * История диалогов хранится в **PostgreSQL**
    * На данный момент реализованы и опробованы подходы
      * Скользящее окно -- хранятся последние пять пар реплик
    * есть команда `/newchat` для сброса контекста

## 🎛 CLI Команды (Управление RAG-пайплайном)

Вы можете управлять процессами индексации и тестировать качество ответов вручную через консоль, используя сервис `rag-cli`.

### 1. Индексация (Сбор данных)
Заполняет векторную базу данных информацией с сайта.
**Результат:** Создаются индексы (ChromaDB + BM25), а полный список созданных текстовых чанков сохраняется в файл `output/indexing.txt` для ручной проверки.

*   **Быстрая индексация (Тестовый режим):**
    Обрабатывает только URL, указанные в `config.yaml` в секции `test_urls`. Идеально для проверки парсера.
    ```bash
    docker-compose exec rag-cli python main.py index test
    ```

*   **Полная индексация (Prod режим):**
    Запускает краулер, который обходит весь сайт (на глубину `max_depth` из конфига) и индексирует все найденные страницы.
    ```bash
    docker-compose exec rag-cli python main.py index
    ```

### 2. Тестирование поиска (Retrieve)
Проверяет, какие документы (чанки) система находит для ответа, но **не генерирует** сам ответ через LLM. Полезно для отладки ретривера.

*   **Поиск по конкретному вопросу:**
    Выводит найденные документы прямо в консоль.
    ```bash
    docker-compose exec rag-cli python main.py retrieve -q "Кто декан факультета?"
    ```

*   **Пакетное тестирование (Evaluation):**
    Прогоняет поиск по всем вопросам из файла `qa-test-set.yaml`. Результаты сохраняются в файл `output/retriever.txt`.
    ```bash
    docker-compose exec rag-cli python main.py retrieve
    ```

### 3. Тестирование полного цикла (Answer)
Запускает полный RAG-пайплайн: Поиск + Генерация ответа через LLM.

*   **Ответ на конкретный вопрос:**
    Генерирует ответ и выводит его в консоль.
    ```bash
    docker-compose exec rag-cli python main.py answer -q "Какие проходные баллы были в 2025 году?"
    ```

*   **Пакетное тестирование (Evaluation):**
    Генерирует ответы на все вопросы из `qa-test-set.yaml`.
    Эта команда проходит по типам вопросов, указанных в файле `config/config.yaml` 
    в свойстве `evaluation_settings.mode:` "all", "questions", "scenarios"
    ```bash
    docker-compose exec rag-cli python main.py answer
    ```
    
    Если вам нужен конкретный тип -- укажите его последним в команде
    ```bash
    docker-compose exec rag-cli python main.py answer all # questions, scenarios
    ```

    Результаты (вопрос, ожидаемый ответ, ответ бота) сохраняются в `output/answer.txt`.

## 🧪 Тестирование и Эксперименты

Проект включает встроенный CLI-пайплайн для A/B тестирования различных стратегий RAG.

Есть поддержка двух типов автоматизированных тестов:
- Questions (Single-turn): Проверка точности ответов на одиночные вопросы. (`questions`)
- Scenarios (Multi-turn): Проверка памяти бота. Скрипт проходит по цепочке вопросов (например: «Кто декан?» -> «А в каком он кабинете?»), проверяя способность системы удерживать контекст.

Тесты используют **изолированные базы данных** (например, `data/chroma_db_semantic`)

### 1. Запуск окружения для тестов

Для проведения экспериментов не обязательно запускать Телеграм-бота. Достаточно поднять базу данных, нейросеть и CLI-контейнер.

Выполните команду:
```bash
docker-compose up -d ollama postgres rag-cli --build
```
*(Если это первый запуск, дождитесь загрузки моделей в логах `ollama`)*.

### 2. Запуск пайплайна

Все команды выполняются внутри контейнера `rag-cli`:

```bash
docker-compose exec rag-cli python main.py test [параметры]
```

### Параметры тестирования

| Параметр | Значения | Краткое описание |
| :--- | :--- | :--- |
| `eval_mode` | `all`, `questions`, `scenarios` | **(Позиционный)** Тип тестов. `all` — всё, `questions` — одиночные, `scenarios` — диалоги. |
| `--chunker` | `markdown`, `semantic`, `unstructured` | Стратегия разбиения текста. Влияет на качество контекста. |
| `--retriever` | `hybrid`, `vector` | Алгоритм поиска. Влияет на Hit Rate (точность нахождения). |
| `--index-mode` | `test`, `full` | Объем данных. `test` — по списку из YAML, `full` — обход всего сайта. |
| `--force-index` | Флаг | Принудительная пересборка базы данных с нуля. |

#### Подробное описание параметров
**0. `eval_mode` (Режим тестирования)**
Определяет, какие именно кейсы из `qa-test-set.yaml` будут запущены.
*   **`all` (Default):** Последовательный запуск одиночных вопросов и многошаговых сценариев.
*   **`questions`:** Тестирование классического RAG (вопрос-ответ) без учета истории.
*   **`scenarios`:** Тестирование памяти бота. Проверка того, как LLM переформулирует вопросы и использует контекст предыдущих реплик.

**1. `--chunker` (Алгоритм чанкинга)**
Определяет, как HTML-страницы превращаются в текстовые блоки.
*   **`markdown` (Default):** Конвертирует весь HTML в Markdown, затем бьет по заголовкам (`#`, `##`).
*   **`semantic`:** "Умный" HTML-парсера. Сохраняет иерархию заголовков и превращает строки таблиц (`<tr>`) в отдельные смысловые документы. Рекомендуется для страниц типа "Деканат" или "Расписание".
*   **Важно:** Каждый алгоритм сохраняет данные в свою изолированную папку (например, `data/chroma_db_markdown` или `data/chroma_db_semantic`), поэтому переключение этого параметра не требует удаления предыдущей базы.

**2. `--retriever` (Стратегия поиска)**
*   **`hybrid` (Default):** Комбинация двух вариантов: Векторный поиск (семантика) + BM25 (ключевые слова).
*   **`vector`:** Только векторный поиск (ChromaDB). Полезно для сравнения, чтобы понять, насколько BM25 улучшает (или ухудшает) результаты.

**3. `--index-mode` (Режим источника данных)**
*   **`test` (Default):** Индексирует только URL, указанные в `config.yaml` в секции `test_urls`.
*   **`full`:** Запускает Crawler, который обходит весь сайт (до глубины `max_depth`).

**4. `--force-index` (Принудительная переиндексация)**
*   Если флаг **не установлен**: Скрипт проверит наличие базы для выбранного `--chunker`. Если база есть — она будет использована (экономия времени). Если нет — она будет создана.
*   Если флаг **установлен**: Скрипт **удалит** старую папку с базой для выбранного чанкера и запустит процесс сбора данных заново. Используйте это, если вы изменили логику парсера, настройки модели или список URL.

### Примеры сценариев

**Сценарий А: Быстрая проверка (Markdown)**
Использует существующий индекс (или создает его, если нет) на малом наборе страниц.
```bash
docker-compose exec rag-cli python main.py test --chunker=markdown
```

**Сценарий Б: Сравнение с "Семантическим" чанкером**
Принудительно пересоздает индекс с новым алгоритмом (Semantic Chunking) и прогоняет тесты.
```bash
docker-compose exec rag-cli python main.py test --chunker=semantic --force-index
```

**Сценарий В: Тест на полном объеме данных**
Проверяет, как система ведет себя, если проиндексировать весь сайт целиком.
```bash
docker-compose exec rag-cli python main.py test --chunker=markdown --index-mode=full --force-index
```

**Сценарий Владислава Шумана: Набор параметров, на котором регулярно тестирует Владислав**
```bash
docker-compose exec rag-cli python main.py test --chunker=markdown --retriever=hybrid --index-mode=test --force-index
```

### Матрица сценариев и чекпоинты

Несколько прогонов с разными параметрами ретривера (`search_k`, веса гибрида, `top_n`, `active_type: chroma_bm25 | qdrant`) задаются в [`config/config.yaml`](config/config.yaml) в секции **`evaluation_scenarios`**. Оркестратор — команда **`test-matrix`**, состояние — файл чекпоинта (по умолчанию путь из **`paths.default_checkpoint_path`**, обычно `check_points/default_checkpoint.json`). Каталог `check_points/` в репозитории не хранится (см. `.gitignore`).

#### Предварительные условия

- Запущены контейнеры, как для обычных тестов (`postgres`, `rag-cli`, при необходимости `ollama` или нативная Ollama на хосте).
- Выполнены миграции: `docker-compose exec rag-cli alembic upgrade head`.
- Для сценария с `retrievers.active_type: qdrant` должен быть доступен сервис Qdrant (как в вашем `docker-compose`).

#### Запуск с нуля

Команда перезаписывает чекпоинт новым `run_id` и проходит по всем сценариям из `evaluation_scenarios` по порядку.

```bash
docker-compose exec rag-cli python main.py test-matrix
```

Через интерактивное меню (после шагов 1–3 скрипт предложит меню):

```bash
chmod +x test.sh
./test.sh
# пункт 3 — матрица, пункт 5 — выход из меню
```

#### Продолжение после паузы или обрыва

Состояние (включая частичные результаты и позицию в блоках «одиночные вопросы» и «диалоги») читается из чекпоинта:

```bash
docker-compose exec rag-cli python main.py test-matrix --resume
```

Отдельный скрипт-обёртка (удобно с ноутбука без меню):

```bash
chmod +x resume.sh
./resume.sh
```

В `./test.sh` это соответствует пункту **4**.

#### Пауза между вопросами (graceful)

Пауза срабатывает **только после завершения текущего вопроса или шага диалога** (не обрывает вызов LLM в середине).

1. Файл-флаг: создайте файл по пути **`paths.pause_flag`** из конфигурации (по умолчанию `output/pause.flag`):

   ```bash
   docker-compose exec rag-cli touch output/pause.flag
   ```

2. Либо отправьте процессу **SIGINT / SIGTERM** (например, `Ctrl+C` в сессии, где крутится прогон): обработчик создаёт тот же файл паузы; дальше цикл дорабатывает текущий шаг и выходит, сохранив чекпоинт.

После остановки для продолжения используйте **`test-matrix --resume`** (или `./resume.sh`). При старте матрицы без `--resume` флаг паузы в начале прогона удаляется, чтобы не остановить первый шаг сразу.

#### Где смотреть артефакты

- Лог трассировки и отчёты по-прежнему в **`output/`** (`trace_*.md`, `report_*.md`); для матрицы имя суффикса стабилизируется в рамках сценария, чтобы при resume дописывать один и тот же trace.

#### Индексы Chroma/BM25 и смена backend

Для каждого сценария подставляются пути вида `data/chroma_db_{chunker}` и BM25 с тем же суффиксом. Если сценарии делят один и тот же `chunker` и не включают `force_index: true`, существующий индекс переиспользуется без переиндексации. Смена `active_type` на `qdrant` не удаляет локальные папки Chroma, но поиск идёт в Qdrant (нужна актуальная коллекция по вашей процедуре индексации).

### Метрики качества
В конце прогона выводится отчет:
*   **Hit Rate (Retrieval):** Эвристическая оценка точности поиска (нашел ли ретривер ключевые слова из ответа).
*   **Similarity (Generation):** Оценка смысловой близости ответа бота к эталону (используется модель `rubert-tiny2`).
*   **Latency:** Среднее время генерации ответа.

## Идеи и подходы

Информация от сюда -- это просто мои какие-то находки либо потенциальные идеи. Впоследствии информация от сюда будет удаляться --
либо я применил ту или иную идею И он уже расписана где-то выше, либо я отказался/ удалил эту идею 

- Hypothetical Document Embeddings
    - Вместо того чтобы переписывать вопрос(Query Reformulation), мы попросим LLM сгенерировать выдуманный ответ на основании истории
- Multi-Query Retrieval
  - Если запрос сложный, то можно попросить LLM разбить его на подзапросы
- Semantic Routing (Классификация)
  - Можно также добавить Классификацию вопроса/запроса пользователя. Например
    - Smalltalk (Привет как дела?)
    - RAG (Кто декан?)
    - Schedule (Какая пара у третьего курса?)
- Small-to-Big Retrieval (Parent Document Retrieval)
  - Разбей документ на мелкие чанки (для точного поиска), но храни ссылку на родительский (большой) кусок текста-
- Query Decomposition (Sub-queries)
  - Если вопрос сложный («Сравни выручку А и Б за 5 лет»), LLM разбивает его на под-вопросы, ищет ответы на каждый по отдельности, а потом синтезирует финал
- Reranker (Cross-Encoders)
  - получил 100 документов, и потом отдельной моделью получил 5 лучших
- Self-RAG / Self-Correction
  - Обучи или заставь LLM оценивать свои ответы. «Достаточно ли информации в контексте?», «Нет ли в моем ответе галлюцинаций относительно источника?». Если ответ плохой — иди на второй круг поиска.
- Semantic Caching
  - Используй кеш (например, GPTCache или Redis), чтобы хранить ответы на семантически похожие вопросы.
- Query Routing (Маршрутизация)
  - классификация вопроса (если это просто Привествие -- сюда. если вопрос, то в RAG)
- Reciprocal Rank Fusion (RRF)
  - Если используешь несколько методов поиска (векторный, ключевые слова, поиск по графам), сочетай их результаты с помощью алгоритма RRF
- Auto-Correction запроса 
  - Перед поиском прогони запрос пользователя через маленькую модель для исправления опечаток и нормализации терминов 
- Собирать отзывы о вопросах
- Использовать Qdrant
